Island Cars

During a particularly work-heavy week of the busy CS-filled semester, I impulsively bought flights to Kauai for the moment finals ended. While searching for a good car rental agency on this tropical paradise of an island, I came across this site: http://www.islandcars.net. The colors are bland, the font is outdated, and the links have been promising improvements for the last 13 years. If I had stumbled upon that page unknowingly I would have assumed the Island Cars business had bit the dust decades ago. Upon inspection the header tag revealed that the last update to this site was performed on May 20, 2003. It is about time for an update. Chris and I decided to redesign the entire site as a challenge for our final project. We created the main page with Bootstrap to give it a professional feel. Users can check certain dates for vehicle availability. We allowed reservations to be made on the site that automatically store information in a database. An administrator may securely log in to manage the vehicle fleet and process reservations. These improvements provide much of the functionality that we would expect to see from a rental agency website. It is possible that the information we display on the website and the ability to make reservations could save the agency from hiring too many people to perform those simple tasks. With a professional and useful website completed, Chris and I are proud to display our work for future employers. This site has the potential to become the live website for Island Cars of Kauai if they wanted to tweak it to make it their own.

V4

Product Description: My Progress Tracker now lets a user view a homepage with forms that are held in a database. New users and progress forms can be created and added to the database.

Form Validation: To create a new user requires a few textboxes that need form validation. I relied upon Chrome's "required" keyword in html that will pull up warning box if a text entry is not in the right format. This feature would not work for Safari. I used the built-in HTML5 validations that are quite useful. Creating a new progress form was a different matter. I decided to only use selection boxes and checkboxes to make it impossible for the user to submit erroneous data.

Design Decisions: The majority of my time was spent fixing the database so that user and form data stayed consistent across all tables. I did not find it necessary to update the new forms or edit old forms. This task seemed unnecessary when compared with the core functionality of the progress tracker.

Testing: I started my design using a hard coded student profile. Eventually I slowly worked my way into having each page custom built to fit whichever student was logged in. There are more than enough test data in my database.

References: This assignment was accomplished with inspiration from Jim's code. There are tiny snippets of html validation that are pulled from W3 Schools.
